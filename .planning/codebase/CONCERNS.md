## Tech Debt

**Duplicate application entry points (web_app.py vs api.py):**
- Issue: Two nearly identical FastAPI applications exist side by side. `web_app.py` (823 lines) and `api.py` (796 lines) both define the same Pydantic models (`LoadRepositoryRequest`, `QueryRequest`, `QueryResponse`, etc.) and expose the same endpoints (`/query`, `/query-stream`, `/load`, `/index`, `/sessions`, etc.)
- Files: `/Users/knakanishi/Repositories/FastCode/web_app.py`, `/Users/knakanishi/Repositories/FastCode/api.py`
- Impact: Bug fixes and new endpoints must be applied twice; the two implementations will inevitably diverge. It is unclear which entry point is canonical for production.
- Fix approach: Consolidate into a single FastAPI app. One of the files should be deleted; the survivor should serve static HTML directly (as `web_app.py` does) or serve as a pure API with `web_interface.html` served separately.

**Commented-out query cache logic:**
- Issue: Query result caching is intentionally disabled with a large block of commented-out code and the comment "NOTE: Query result caching is disabled to ensure full iterative_agent flow". The original cache path is preserved as dead comments (lines 310-323 in `fastcode/main.py`).
- Files: `/Users/knakanishi/Repositories/FastCode/fastcode/main.py` lines 310–324, 418–421
- Impact: Every query, even repeated identical ones, runs the full iterative agent loop. No performance benefit from caching, and the commented block causes confusion about intent.
- Fix approach: Either remove the dead comment block if caching is truly abandoned, or implement a cache invalidation strategy that is compatible with the iterative agent.

**Commented-out temp directory for cloned repos:**
- Issue: `RepositoryLoader.load_from_url()` has a commented-out line (`# self.temp_dir = tempfile.mkdtemp(...)`) that was replaced with hardcoded `./repos`. Cloned repositories are never cleaned up automatically.
- Files: `/Users/knakanishi/Repositories/FastCode/fastcode/loader.py` line 58
- Impact: Disk usage grows indefinitely with each new repository cloned. The commented code leaves implementation intent unclear.
- Fix approach: Implement explicit cleanup in `cleanup()` or on re-clone; document the lifecycle decision.

**TODO stub: multi-repo graph merge:**
- Issue: Line 1195 of `fastcode/main.py` contains `# TODO: Merge additional repository graphs if needed` immediately after the graph merge logic in `_load_multi_repo_cache()`. The preceding code already calls `merge_from_file` but the TODO signals the path is incomplete.

**Hardcoded default model in `_get_default_config()`:**

## Known Bugs

**`embed_code_elements` uses `list.index()` — O(n²) and wrong for duplicates:**
- Symptoms: When multiple code elements produce the same text (duplicated function names, boilerplate), `elements.index(elem)` returns the index of the *first* matching element, assigning the wrong `embedding_text` to subsequent elements.
- Files: `/Users/knakanishi/Repositories/FastCode/fastcode/embedder.py` line 107
- Trigger: Any two code elements with identical `_prepare_code_text` output in the same batch.
- Workaround: None; silently corrupts embedding metadata for duplicates.

**`delete_by_filter` in `VectorStore` rebuilds metadata but not the FAISS index:**
- Symptoms: After calling `delete_by_filter`, `self.metadata` is pruned but the FAISS index still contains the deleted vectors. Subsequent searches can return results for metadata that no longer exists (index/metadata mismatch).
- Files: `/Users/knakanishi/Repositories/FastCode/fastcode/vector_store.py` lines 610–645
- Trigger: Any code path that calls `delete_by_filter` (e.g. repository deletion flow).
- Workaround: The method logs a warning about this limitation but does not prevent the mismatch.

**`search_repository_overviews` prints debug output to stdout in production:**
- Symptoms: Every call to `search_repository_overviews` unconditionally executes `print(f"similarity: {similarity}, repo_name: {repo_name}")` regardless of log level.
- Files: `/Users/knakanishi/Repositories/FastCode/fastcode/vector_store.py` line 339
- Trigger: Any multi-repo query that calls the overview search path.
- Workaround: None; stdout is polluted in all deployments including Docker.

**`_select_relevant_repositories_by_embedding` prints to stdout:**
- Symptoms: Two `print()` calls on lines 504 and 513 of `fastcode/retriever.py` write repo selection scores directly to stdout, bypassing the logging framework.
- Files: `/Users/knakanishi/Repositories/FastCode/fastcode/retriever.py` lines 504, 513
- Trigger: Multi-repo queries using the embedding-based repository selection path.
- Workaround: None.

**`query_processor.py` prints LLM response to stdout unconditionally:**
- Symptoms: `print(f"LLM response of _enhance_with_llm: {response}")` on line 536 of `fastcode/query_processor.py` leaks full LLM response content (potentially sensitive) to stdout.
- Files: `/Users/knakanishi/Repositories/FastCode/fastcode/query_processor.py` line 536
- Trigger: Any query that uses LLM enhancement (default in `config.yaml`: `llm_enhancement_mode: "always"`).
- Workaround: None.

## Security Considerations

**CORS wildcard allows all origins:**
- Risk: Both `api.py` and `web_app.py` configure CORS with `allow_origins=["*"]`, `allow_credentials=True`, and `allow_methods=["*"]`. Combining wildcard origins with `allow_credentials=True` is dangerous: any website can make authenticated cross-origin requests to the API.
- Files: `/Users/knakanishi/Repositories/FastCode/api.py` lines 105–111, `/Users/knakanishi/Repositories/FastCode/web_app.py`
- Current mitigation: None; the API is typically run locally, but Docker deployment exposes it on `0.0.0.0:8000`.
- Recommendations: Set `allow_origins` to an explicit list of trusted origins and set `allow_credentials=False` unless session cookies are actually used.

**No authentication on any API endpoint:**
- Risk: All FastAPI endpoints (`/query`, `/delete-repos`, `/upload-zip`, `/sessions`, etc.) are completely unauthenticated. Anyone with network access to port 8000 can submit queries, delete repositories, and upload arbitrary ZIP archives.
- Files: `/Users/knakanishi/Repositories/FastCode/api.py`, `/Users/knakanishi/Repositories/FastCode/web_app.py`
- Current mitigation: None. Production deployments rely entirely on network-level access control.
- Recommendations: Add at minimum a static API key header check (`X-API-Key`) via FastAPI middleware or `Depends()`.

**ZIP file upload does not validate archive contents:**
- Risk: The `/upload-zip` endpoint (`api.py` lines 339–415) writes ZIP contents directly to disk without validating paths inside the archive. A crafted ZIP (zip slip attack) could write files outside the `./repos/` directory by using `../` paths in archive member names.
- Files: `/Users/knakanishi/Repositories/FastCode/api.py` lines 382–392
- Current mitigation: The extracted top-level directory is moved via `shutil.move`, which partially limits exposure, but member-level path traversal is not checked.
- Recommendations: Use `zipfile.ZipFile.namelist()` to validate all member paths before extraction; reject any archive containing absolute paths or `..` components.

**LLM responses are printed to stdout in production (data leakage):**
- Risk: Debug `print()` calls in `fastcode/query_processor.py` (line 536) output full LLM responses — which contain source code retrieved from indexed repositories — to stdout. In containerized deployments, stdout is captured in logs that may be accessible to multiple parties.
- Files: `/Users/knakanishi/Repositories/FastCode/fastcode/query_processor.py` line 536
- Current mitigation: None.
- Recommendations: Replace all `print()` calls in the `fastcode/` package with `self.logger.debug()` calls.

**API keys read via `os.getenv()` scattered across many modules without central validation:**
- Risk: `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `MODEL`, and `BASE_URL` are read independently in `fastcode/answer_generator.py`, `fastcode/iterative_agent.py`, `fastcode/query_processor.py`, `fastcode/repo_overview.py`, and `fastcode/repo_selector.py`. Missing keys produce only a `logger.warning`; the client is still constructed with `api_key=None` and will fail at the first actual API call.
- Files: Multiple files in `/Users/knakanishi/Repositories/FastCode/fastcode/`
- Current mitigation: Warning log at initialization; hard failure deferred until query time.
- Recommendations: Centralize credential loading in a single config/secrets module; raise at startup if required keys are missing.

**Nanobot shell execution guard uses regex on lowercased command:**
- Risk: `ExecTool._guard_command()` applies deny-list patterns against `cmd.lower()`. A command like `RM -RF /` passes because the pattern `r"\brm\s+-[rf]{1,2}\b"` matches `rm`, but case normalization occurs after tokenization, so `RM -RF /` becomes `rm -rf /` after lower — this actually works. However, the `path traversal` check only looks for `../` and `..\` as string literals, not URL-encoded variants (`%2e%2e%2f`) or shell expansion tricks (`$'../'`).
- Files: `/Users/knakanishi/Repositories/FastCode/nanobot/nanobot/agent/tools/shell.py` lines 111–141
- Current mitigation: Basic regex deny-list; `restrict_to_workspace` flag defaults to `False`.
- Recommendations: Enable `restrict_to_workspace=True` by default; add URL-decode normalization before guard checks.

## Performance Bottlenecks

**`merge_from_index` reconstructs vectors one-by-one with a Python loop:**
- Problem: `VectorStore.merge_from_index()` calls `other_index.reconstruct(int(i), vectors[i])` in a Python `for` loop (lines 594–595), iterating over every vector in the source index. For large repositories (tens of thousands of vectors), this is extremely slow.
- Files: `/Users/knakanishi/Repositories/FastCode/fastcode/vector_store.py` lines 589–598
- Cause: FAISS `reconstruct_n` batch API is not used.
- Improvement path: Replace the loop with `other_index.reconstruct_n(0, n_vectors, vectors)` for a single batch reconstruction call.

**`scan_available_indexes` still deserializes entire metadata pickle per repo:**
- Problem: Although sampling is applied (lines 704–720), the entire pickle file is deserialized into memory before sampling (`data = pickle.load(f)` on line 699). For a large repository with 100k+ elements, this reads and parses the full pickle even though only the first 100 entries are used.
- Files: `/Users/knakanishi/Repositories/FastCode/fastcode/vector_store.py` lines 697–720
- Cause: Python pickle format requires full deserialization; no streaming support.
- Improvement path: Store a separate lightweight `{repo_name}_summary.json` at index-save time containing element count, file count, and URL — loaded instead of the full pickle during scans.

**`_select_relevant_repositories_by_embedding` calls `load_repo_overviews()` on every query:**
- Problem: `VectorStore.load_repo_overviews()` reads and deserializes `repo_overviews.pkl` from disk on every invocation. In multi-repo mode this pickle is loaded for every query and every SSE chunk.
- Files: `/Users/knakanishi/Repositories/FastCode/fastcode/vector_store.py` line 310 (inside `search_repository_overviews`), `/Users/knakanishi/Repositories/FastCode/fastcode/retriever.py`
- Cause: No in-memory cache of the overviews dict between calls.
- Improvement path: Cache the deserialized overviews in `VectorStore` and invalidate on `save_repo_overview()` and `delete_repo_overview()`.

**`embed_code_elements` uses `list.index()` — O(n²) embedding text assignment:**
- Problem: After batch embedding, the code assigns `embedding_text` by calling `texts[elements.index(elem)]`, which does a linear scan through `elements` for each item — O(n²) overall for a batch of n elements.
- Files: `/Users/knakanishi/Repositories/FastCode/fastcode/embedder.py` line 107
- Cause: Identity-based index lookup used instead of positional zip.
- Improvement path: Replace with `for i, (elem, embedding) in enumerate(zip(elements, embeddings)):` and use `texts[i]` directly.

## Fragile Areas

**Global singleton `fastcode_instance` with no concurrency protection:**
- Files: `/Users/knakanishi/Repositories/FastCode/api.py` line 114, `/Users/knakanishi/Repositories/FastCode/web_app.py`
- Why fragile: `fastcode_instance` is a module-level global mutated by lazy initialization (`_ensure_fastcode_initialized()`) and by the `/repository` DELETE endpoint (which calls `FastCode()`). Multiple concurrent requests can race on initialization, and `_ensure_fastcode_initialized()` is not guarded by an `asyncio.Lock` or threading lock.
- Safe modification: Wrap the lazy init with `asyncio.Lock`; ensure the reset path in `DELETE /repository` is serialized relative to active queries.
- Test coverage: No tests exist for concurrent initialization.

**`VectorStore.delete_by_filter` metadata/index desync:**
- Files: `/Users/knakanishi/Repositories/FastCode/fastcode/vector_store.py` lines 610–645
- Why fragile: The method prunes `self.metadata` but leaves the FAISS index intact, resulting in an inconsistent state. Any search after deletion will return indices into the old (larger) FAISS index, mismatching the pruned metadata list and causing `IndexError` or silently returning stale data.
- Safe modification: Do not call `delete_by_filter` for any real deletion path until it is reimplemented to rebuild the FAISS index. Use `remove_repository()` (which saves a fresh index) instead.
- Test coverage: None detected.

**`IterativeAgent` carries mutable shared state across requests:**
- Files: `/Users/knakanishi/Repositories/FastCode/fastcode/iterative_agent.py`
- Why fragile: `self.iteration_history`, `self.tool_call_history`, and `self.dialogue_history` are reset at the start of each `retrieve_with_iteration()` call but are instance attributes. If two requests share the same `IterativeAgent` instance concurrently (e.g., two parallel API calls), these lists will be corrupted. The `IterativeAgent` is created once inside `HybridRetriever.__init__` and reused for all requests.
- Safe modification: Either make the agent stateless (pass state as local variables) or instantiate a fresh agent per request.
- Test coverage: None detected.

**DingTalk channel uses `type: ignore` to silently swallow missing optional dependency:**
- Files: `/Users/knakanishi/Repositories/FastCode/nanobot/nanobot/channels/dingtalk.py` lines 30–33
- Why fragile: Four class-level assignments (`CallbackHandler = object`, `CallbackMessage = None`, etc.) are marked `# type: ignore[assignment,misc]` to suppress mypy errors when the `dingtalk-stream` package is not installed. Code that checks `if CallbackMessage is None` will silently do nothing instead of raising a clear configuration error.
- Safe modification: Raise `ImportError` with a clear message at channel startup if the dependency is missing, rather than substituting `None` sentinel values.
- Test coverage: None detected.

## Scaling Limits

**In-memory metadata list grows with every indexed element:**
- Current capacity: `VectorStore.metadata` is a plain Python `list[dict]`. A large repository with 50k elements, each with a 500-byte metadata dict, consumes ~25 MB in memory. In multi-repo mode with 10 repositories this becomes ~250 MB, all resident for the lifetime of the process.
- Limit: Memory exhaustion above approximately 200+ repositories or very large monorepo indexes.
- Scaling path: Store metadata in a SQLite sidecar or a memory-mapped structure; load only what is needed for a given query's result set.

**BM25 index held fully in memory as Python lists:**
- Current capacity: `full_bm25_elements` and `full_bm25_corpus` in `HybridRetriever` hold all indexed code element text as Python lists. `rank_bm25` operates entirely in RAM.
- Limit: 100k+ elements will consume several GB of RAM.
- Scaling path: Switch to a disk-backed inverted index (e.g. Whoosh, Tantivy) for large-scale deployments.

## Dependencies at Risk

**`faiss-cpu` pin-free installation:**
- Risk: `faiss-cpu` has a history of binary compatibility issues across Python versions and operating systems. Without a pinned version, upgrades may silently change HNSW behavior or introduce ABI incompatibilities.
- Impact: Index corruption or segfaults on upgrade.
- Migration plan: Pin to `faiss-cpu==1.8.0` (or the current tested version); test on upgrade.

**`chromadb` in requirements but not used:**
- Risk: `chromadb` is a large package (~500MB installed) that pulls in transitive dependencies including `onnxruntime`. It is listed in `requirements.txt` but no code in the project imports it.
- Impact: Unnecessary installation size, transitive security vulnerabilities.
- Migration plan: Remove from `requirements.txt`.

**`dingtalk-stream` optional dependency not declared:**
- Risk: The DingTalk channel silently degrades when `dingtalk-stream` is not installed (type: ignore workaround). The package is not listed in `nanobot/pyproject.toml` as an optional extra.
- Impact: Users enabling DingTalk discover the failure at runtime, not install time.
- Migration plan: Add `dingtalk-stream` as an optional extra in `nanobot/pyproject.toml` and document it.
