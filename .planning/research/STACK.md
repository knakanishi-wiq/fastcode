## Recommended Stack

### Core Technologies

| Technology | Version | Purpose | Why Recommended |
|------------|---------|---------|-----------------|
| `uv` | `0.10.6` (latest as of 2026-02-26) | Package manager, lock file generation, Docker installation | Astral's official tool — fastest Python resolver, native lockfile format, official Docker image for Dockerfile `COPY --from` pattern, handles both runtime and dev groups cleanly |
| `pyproject.toml` | PEP 517/621 standard | Project metadata + runtime dependencies | The modern standard; uv generates and manages this natively |
| `uv.lock` | uv lockfile format | Reproducible installs across environments | Universal lockfile (cross-platform resolution); generated by `uv sync` or `uv lock` |

### Build Backend Decision

**Recommendation: No build backend (plain application mode)**

FastCode is an application, not a published library. `uv init` (plain app) intentionally omits a `[build-system]` table — the project is not installed into the venv as a package, it runs in-place via `uv run` or directly with the activated `.venv`.

This means:

- No `hatchling`, `setuptools`, or `uv_build` required
- `uv sync --locked` installs dependencies only, not the project itself
- `fastcode/` code is imported normally from the working directory (same as now)
- CLI entry points (`fastcode.cli:main`) work via `uv run python -m fastcode.cli` or PATH after `uv sync`

**When to use a build backend instead:**

| Condition | Backend | Reason |
|-----------|---------|--------|
| Need `uv run fastcode` CLI entrypoint | `hatchling` or `uv_build` | Requires `[project.scripts]` and package installation |
| Publishing to PyPI | `hatchling` | Most widely supported for pure Python |
| Already using src/ layout | `hatchling` or `uv_build` | Both handle src-layout equally well |

For FastCode (app with `fastcode/` flat layout, not published), the no-build-backend path is cleanest.

### Dependency Groups

| Group | pyproject.toml section | Sync behavior | Purpose |
|-------|----------------------|---------------|---------|
| Runtime | `[project.dependencies]` | Always installed | All current requirements.txt entries except pytest/pytest-asyncio/pytest-cov |
| Dev/test | `[dependency-groups]` `dev` | Default (installed locally) | pytest, pytest-asyncio, pytest-cov |

Use `[dependency-groups]` (PEP 735, uv-native) rather than `[project.optional-dependencies]` for dev/test because:

- Dev deps are NOT published (FastCode is not a PyPI package)
- `[dependency-groups]` is the current uv-recommended approach (uv docs explicitly note `tool.uv.dev-dependencies` is deprecated)
- `uv sync --no-dev` (or `UV_NO_DEV=1`) skips the dev group in Docker

## Dockerfile Pattern

### Recommended: Single-Stage with Layer Caching

FastCode's Dockerfile is already single-stage (not multi-stage). The recommended uv pattern for single-stage is:

```dockerfile
FROM python:3.12-slim-bookworm

# Install system deps (unchanged from current Dockerfile)
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        git \
        build-essential \
        curl \
        ca-certificates && \
    apt-get autoremove -y && \
    rm -rf /var/lib/apt/lists/*

# Install uv (pin to exact version for reproducibility)
COPY --from=ghcr.io/astral-sh/uv:0.10.6 /uv /uvx /bin/

WORKDIR /app

# Install dependencies first (cached layer — only re-runs when lock/pyproject change)
ENV UV_COMPILE_BYTECODE=1 \
    UV_LINK_MODE=copy \
    UV_NO_DEV=1

RUN --mount=type=cache,target=/root/.cache/uv \
    --mount=type=bind,source=uv.lock,target=uv.lock \
    --mount=type=bind,source=pyproject.toml,target=pyproject.toml \
    uv sync --locked --no-install-project

# Copy application code
COPY fastcode/ fastcode/
COPY api.py ./
COPY config/ config/

# Activate venv for CMD
ENV PATH="/app/.venv/bin:$PATH"

EXPOSE 8001
ENV PYTHONUNBUFFERED=1

CMD ["python", "api.py", "--host", "0.0.0.0", "--port", "8001"]
```

Key decisions in this pattern:

- `--mount=type=bind` for `uv.lock` and `pyproject.toml` — installs deps without COPYing the files first, keeping them out of an early layer
- `--no-install-project` — installs only dependencies, not the FastCode project itself (consistent with no-build-backend choice)
- `UV_NO_DEV=1` — skips dev group (pytest etc.) in production image
- `UV_COMPILE_BYTECODE=1` — compiles `.pyc` at build time for faster startup
- `UV_LINK_MODE=copy` — silences cross-filesystem link warnings in Docker
- `COPY --from=ghcr.io/astral-sh/uv:0.10.6` — pins uv to an exact version, not `:latest`

### Multi-Stage Alternative (NOT recommended for FastCode)

Multi-stage builds copy only `.venv` into the final image, excluding source code. FastCode's source code is not sensitive and the complexity cost (two base images, COPY --from=builder paths) is not worth it for an internal tool.

Use multi-stage if: FastCode source code must be excluded from the shipped image for IP reasons.

## Version Compatibility

| Package | Compatible With | Notes |
|---------|-----------------|-------|
| `uv 0.10.6` | Python `>=3.12` | FastCode uses `python:3.12-slim-bookworm` base image; uv has no issue with 3.12 or 3.13 |
| `uv.lock` | `uv >=0.10.x` | Lock format is versioned; generated lock is pinned to the uv version that produced it — regenerate if uv is upgraded |
| `--mount=type=cache` | Docker BuildKit | Requires `DOCKER_BUILDKIT=1` or Docker 23+ (BuildKit on by default since Docker 23). `docker-compose build` enables BuildKit automatically in Compose v2 |

## What NOT to Use

| Avoid | Why | Use Instead |
|-------|-----|-------------|
| `pip install -r requirements.txt` in Dockerfile | Slow, no layer-cache optimization for deps vs. code changes | `uv sync --locked --no-install-project` with `--mount=type=bind` |
| `hatchling` or `setuptools` build backend | FastCode is an app, not a library — adding a build backend requires a `src/` layout reorganization or package naming that adds zero value here | No `[build-system]` table |
| `[tool.uv.dev-dependencies]` (legacy uv field) | uv docs explicitly note this field is deprecated in favor of `[dependency-groups]` | `[dependency-groups]` with PEP 735 |
| `COPY --from=ghcr.io/astral-sh/uv:latest` | `:latest` tag breaks reproducibility — future uv versions may change behavior | Pin to `ghcr.io/astral-sh/uv:0.10.6` |
| Multi-stage build | Complexity without benefit — FastCode source isn't sensitive, single-stage is simpler and sufficient | Single-stage with `--mount=type=cache` |
| `ENV TOKENIZERS_PARALLELISM=false` | Dead env var post sentence-transformers removal (v1.1) — this is v1.2 cleanup scope | Remove it |

## Installation Commands

```bash
# Install uv (one-time, local dev)
curl -LsSf https://astral.sh/uv/install.sh | sh

# Initialize project (from repo root, after creating pyproject.toml)
uv sync         # Installs all deps + dev group; generates .venv and uv.lock

# Runtime only (no dev deps)
uv sync --no-dev

# Add a new runtime dep
uv add <package>

# Add a dev dep
uv add --dev <package>

# Update lockfile after manual pyproject.toml edit
uv lock

# Run with uv-managed env
uv run python api.py
```

## Alternatives Considered

| Recommended | Alternative | When to Use Alternative |
|-------------|-------------|-------------------------|
| `uv` | `poetry` | Poetry has broader ecosystem support and a longer track record, but uv is 10-100x faster at resolution and is now the community-recommended default for new projects. For FastCode's simple dep graph, uv's speed advantage is clear. |
| `uv` | `pip-tools` (pip-compile) | pip-tools generates `requirements.txt` lockfiles and is extremely stable. Use it if the team needs to stay fully pip-compatible without adopting a new tool. uv also supports `uv pip compile` as a pip-tools drop-in if needed during transition. |
| No build backend | `hatchling` | Use hatchling if FastCode needs `[project.scripts]` CLI entrypoints (e.g., `fastcode` as an installed command) or if the project moves to a `src/` layout. Current flat layout + direct `python api.py` invocation doesn't need it. |
| `[dependency-groups]` for dev | `[project.optional-dependencies]` extras | Optional deps are correct when dev tooling needs to be installable by external consumers (`pip install fastcode[dev]`). FastCode is not published, so dev extras would never be used externally. |
