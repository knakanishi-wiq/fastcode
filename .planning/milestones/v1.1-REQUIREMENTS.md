# Requirements — v1.1 VertexAI Embedding Migration

**Milestone:** v1.1
**Status:** Defining
**Last updated:** 2026-02-25

---

## Goal

Replace the local sentence-transformers embedding backend with `litellm.embedding()` calling `vertex_ai/gemini-embedding-001` via ADC, completing the GCP-native routing story for both LLM generation (v1.0) and embeddings (v1.1).

---

## Scope

### In Scope

**Functional migration:**
- Rewrite `fastcode/embedder.py` `CodeEmbedder` to call `litellm.embedding()` instead of `SentenceTransformer`
- Pass `task_type="RETRIEVAL_DOCUMENT"` when embedding documents (index time) and `task_type="RETRIEVAL_QUERY"` when embedding queries
- Preserve the existing public API: `embed_text()`, `embed_batch()`, `embed_code_elements()`, `compute_similarity()`, `compute_similarities()`
- Add `task_type` parameter (default `"RETRIEVAL_QUERY"`) to `embed_text()` and `embed_batch()`; update `embed_code_elements()` to pass `"RETRIEVAL_DOCUMENT"`
- Update `fastcode/indexer.py` line 369 (repo overview embedding) to pass `task_type="RETRIEVAL_DOCUMENT"` to `embed_text()`
- Preserve L2 normalization on embedding vectors (FAISS cosine similarity depends on this)

**Configuration:**
- Update `config/config.yaml` embedding section: new model name, add `embedding_dim: 3072`, remove `device` and `max_seq_length` (server-side now)
- Keep `batch_size` and `normalize_embeddings` config keys

**Dependency cleanup:**
- Remove `sentence-transformers` from `requirements.txt`
- Remove Dockerfile pre-bake line (`SentenceTransformer(...)` at build time)
- Update `main.py` default model string (from `sentence-transformers/all-MiniLM-L6-v2` → `vertex_ai/gemini-embedding-001`)

**Validation:**
- Add embedding smoke test: skips when `VERTEXAI_PROJECT` unset (same pattern as v1.0 LLM smoke test)
- Smoke test asserts: embedding returned, correct dimension (3072), vector is L2-normalized

### Out of Scope

- Retrieval strategy changes (FAISS/BM25/graph weights unchanged)
- Async embedding (`litellm.aembedding()`) — FastCode is sync-only
- Multiple embedding providers simultaneously
- Re-indexing of existing repos (expected consequence of dimension change; documented only)
- `output_dimensionality` config knob (use default 3072; can be added in v1.2 if needed)
- ChromaDB or other vector store backends

## Detailed Requirements

### R1 — CodeEmbedder backend replaces sentence-transformers with litellm

`fastcode/embedder.py` — `CodeEmbedder.__init__()` removes all sentence-transformers/torch imports and model loading. Replaces with:
```python

self.embedding_dim = self.embedding_config.get("embedding_dim", 3072)
```
No model download at init. No device detection. No `_load_model()` method.

**Verify:** `import fastcode.embedder` does not import torch or sentence_transformers.

---

### R2 — `embed_batch()` calls `litellm.embedding()` with task_type

```python
def embed_batch(self, texts: List[str], task_type: str = "RETRIEVAL_QUERY") -> np.ndarray:
```

Implementation:
1. Calls `litellm.embedding(model=self.model_name, input=texts, task_type=task_type)`
2. Extracts vectors: `[item["embedding"] for item in response.data]`
3. Converts to `np.ndarray` of shape `(len(texts), embedding_dim)`
4. If `self.normalize`: L2-normalize each vector with `v / np.linalg.norm(v)`
5. Handles `tqdm` progress bar when `len(texts) > 100` (replaces `show_progress_bar` from sentence-transformers)

**Verify:** Returns ndarray shape `(N, 3072)` for N texts.

### R3 — `embed_text()` passes task_type through to `embed_batch()`

## Known Consequences (Not Requirements)

**Existing FAISS indexes are incompatible after this migration.** Dimension changes from 384 → 3072. Any `.index` files and associated pickle files in `./data/vector_store/` must be deleted before first use. FastCode's existing index-loading logic will fail with a dimension mismatch error if stale indexes remain. This is expected and documented; no code change required.

**Batch API behavior with gemini-embedding-001:** litellm uses the Vertex AI `batchEmbedContents` endpoint internally. If the API rejects batches larger than 1, litellm's batch handling may need to be tested. `batch_size: 32` in config is preserved as the outer loop chunk size; internal API behaviour is litellm's responsibility.
