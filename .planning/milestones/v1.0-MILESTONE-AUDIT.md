### Tech Debt Findings

#### 1. Stale default config in `main.py:_get_default_config()`
- **File:** `fastcode/main.py` lines 865-870
- **Issue:** Fallback dict contains `"provider": "openai"` and `"model": "gpt-4-turbo-preview"` — remnant of pre-migration dispatch
- **Impact:** Non-breaking. `AnswerGenerator` ignores `gen_config.get("model")`, reads MODEL env var instead. Only affects temperature/max_tokens in no-config-file code paths (0.1 vs 0.4, 2000 vs 20000 divergence from config.yaml)
- **Severity:** Low

#### 2. Dual model env vars (MODEL vs LITELLM_MODEL)
- **Issue:** AnswerGenerator checks MODEL then falls back to DEFAULT_MODEL (LITELLM_MODEL). Other callers use LITELLM_MODEL only. No documentation note about independence
- **Impact:** Operationally confusing but functionally correct. Both vars documented in .env.example
- **Severity:** Low

#### 3. `truncate_to_tokens` uses OpenAI tokenizer for Gemini models
- **File:** `fastcode/utils.py` — `truncate_to_tokens()`, called by `answer_generator._truncate_context()`
- **Issue:** tiktoken falls back to cl100k_base for `vertex_ai/` model names. Truncation works but token count diverges from Gemini's tokenizer
- **Severity:** Low — functional gap only in truncation accuracy

#### 4. INFRA-04 passive (no explicit retry config)
- **Issue:** No `litellm.num_retries` configured. Library defaults apply silently
- **Severity:** Low
