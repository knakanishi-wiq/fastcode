### E2E Flow: file-on-disk → FTS5-queryable

**Status: COMPLETE**

```
file-on-disk
  → os.walk() + pathspec .gitignore filter
  → CodeParser.parse_file() → FileParseResult (classes, functions, whole-file fallback)
  → index_repo(): mtime_ns fast-path → SHA-256 comparison → skip / DELETE+INSERT
  → INSERT INTO chunks (source_path, content, content_hash, chunk_index, start_offset, end_offset)
  → chunks_ai trigger fires → INSERT INTO chunks_fts(rowid, content)
  → SELECT … FROM chunks_fts JOIN chunks WHERE chunks_fts MATCH ? ORDER BY bm25(chunks_fts) ← verified live
```

Integration checker verified the full chain live:
- `index_repo()` returns `{indexed: 1, skipped: 0, deleted: 0, errors: 0}` for a fresh file
- `sources` row populated with correct `path`, `mtime_ns`, `size`
- `chunks` rows populated (e.g. 2 chunks for a file with 1 class + 1 function)
- FTS5 MATCH query with `bm25()` scoring returns correct results immediately after `index_repo()`

### Phase 13 Readiness

The following query contract for Phase 13 (BM25 retriever) was verified against the live schema today:

```sql
SELECT c.id, c.source_path, c.content, c.chunk_index, c.start_offset, c.end_offset,
       bm25(chunks_fts) as bm25_score
FROM chunks_fts
JOIN chunks c ON c.id = chunks_fts.rowid
WHERE chunks_fts MATCH ?
ORDER BY bm25_score
LIMIT 10
```

All columns are present and return correct values. Phase 13 can also use direct content retrieval (no JOIN) because `chunks_fts` uses `content=chunks` (content-linked).

The existing `fastcode/retriever.py` (in-memory BM25Okapi) has no references to `init_db`, `chunks_fts`, or `index_repo` — zero collision risk when Phase 13 adds the SQLite path.
